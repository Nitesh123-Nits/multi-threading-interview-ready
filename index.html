Thread Creation

All ways to create threads: Thread class, Runnable, Callable, Lambda, FutureTask, Virtual Threads (Java 21)

ThreadRunnableCallableJava 21

🧵 Core Theory — Thread Creation

  • Thread class: Extending Thread is simple but bad practice — Java only allows single inheritance, so extending Thread locks you out of extending any other class.
  • Runnable (preferred): Implementing Runnable separates the task from the thread mechanism. Pass to new Thread(runnable). Safer and more flexible.
  • Callable vs Runnable: Callable<V> returns a value and can throw checked exceptions. Runnable returns void and cannot throw checked exceptions.
  • FutureTask: Wraps a Callable, implements both Runnable (can be passed to Thread) and Future (lets you retrieve the result with .get()).
  • Daemon Threads: Background threads that die automatically when all non-daemon threads finish. JVM does NOT wait for daemons on exit. Must set setDaemon(true) before start().
  • Thread Priority: Hint to the OS scheduler (1=MIN, 5=NORM, 10=MAX). Not guaranteed — OS may ignore it. Do not rely on priority for correctness.
  • Virtual Threads (Java 21): Lightweight JVM-managed threads. Not OS threads. Millions can coexist. Ideal for I/O-bound work. Always daemon. Use Thread.ofVirtual() or Executors.newVirtualThreadPerTaskExecutor().
💡 Interview tip: "Prefer Runnable over Thread extension. Use Callable when you need a return value. Use Virtual Threads for massive I/O concurrency in Java 21."
☕ThreadCreationExample.java
// 1. Extending Thread class
class MyThread extends Thread {
    public MyThread(String name) { super(name); }
    @Override
    public void run() {
        System.out.println("Running: " + getName() + " | Daemon=" + isDaemon());
    }
}

// 2. Implementing Runnable (preferred — single inheritance safe)
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable on: " + Thread.currentThread().getName());
    }
}

// 3. Lambda Runnable
Thread t = new Thread(() -> System.out.println("Lambda thread!"), "LambdaThread");

// 4. Callable — returns value + can throw checked exception
Callable<String> callable = () -> {
    Thread.sleep(500);
    return "Callable Result";
};
FutureTask<String> ft = new FutureTask<>(callable);
new Thread(ft, "CallableThread").start();
String result = ft.get();        // blocks until done
boolean done = ft.isDone();      // true after completion

// 5. Thread Priority & Daemon
Thread low = new Thread(() -> {}, "Low");
low.setPriority(Thread.MIN_PRIORITY);   // 1
low.setPriority(Thread.NORM_PRIORITY);  // 5 (default)
low.setPriority(Thread.MAX_PRIORITY);   // 10
low.setDaemon(true);  // MUST call before start()!
low.start();

// 6. Virtual Threads (Java 21 — lightweight, millions can exist)
Thread vt = Thread.ofVirtual().name("vt-1").start(() ->
    System.out.println("isVirtual: " + Thread.currentThread().isVirtual())
);
vt.join();
// Virtual thread executor:
ExecutorService vtExec = Executors.newVirtualThreadPerTaskExecutor();

Thread Lifecycle & States

NEW → RUNNABLE → BLOCKED / WAITING / TIMED_WAITING → TERMINATED. Key methods: join, sleep, interrupt, yield

Statesjoin()interrupt()

🔄 Core Theory — Thread Lifecycle & States

States

  • NEW: Created, start() not called yet
  • RUNNABLE: Running or ready to run (waiting for CPU)
  • BLOCKED: Waiting to acquire a monitor lock (e.g., entering synchronized block held by another thread)
  • WAITING: Indefinitely waiting — via wait(), join(), park()
  • TIMED_WAITING: Waiting with a timeout — via sleep(), wait(ms), join(ms)
  • TERMINATED: run() completed or threw an exception

Key Methods

  • join(): Current thread waits for the target thread to die. Useful to ensure ordering.
  • sleep(ms): Pauses current thread for ms milliseconds. Releases CPU but does NOT release locks.
  • yield(): Hint to scheduler to give up CPU. No guarantee. Rarely used in production.
  • interrupt(): Sets the interrupt flag. If thread is sleeping/waiting, throws InterruptedException. Always re-set the flag in catch block!
  • isInterrupted(): Checks flag without clearing it.
  • Thread.interrupted(): Checks AND clears the flag.
💡 BLOCKED vs WAITING: BLOCKED = competing for a lock. WAITING = voluntarily waiting for a signal. Critical interview distinction!
☕ThreadLifecycle.java
// Thread States: NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, TERMINATED
Object monitor = new Object();
Thread t = new Thread(() -> {
    synchronized (monitor) {
        try { monitor.wait(500); }    // → TIMED_WAITING (releases lock)
        catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
});

System.out.println(t.getState()); // NEW
t.start();
Thread.sleep(50);
System.out.println(t.getState()); // TIMED_WAITING
t.join();                          // current thread waits for t to die
System.out.println(t.getState()); // TERMINATED

// Key Thread API Methods:
Thread cur = Thread.currentThread();
cur.getName();              // thread name
cur.getId();                // unique thread ID
cur.getPriority();          // 1-10 (5=NORM)
cur.isAlive();              // still running?
cur.isDaemon();             // background thread?
cur.isVirtual();            // Java 21: virtual thread?
cur.isInterrupted();        // check interrupt flag
Thread.interrupted();       // check AND clear interrupt flag

Thread.sleep(1000);         // pause (TIMED_WAITING), releases CPU
Thread.yield();             // hint to scheduler, no guarantee

// Interrupt a sleeping thread:
Thread sleeper = new Thread(() -> {
    try { Thread.sleep(60_000); }
    catch (InterruptedException e) {
        System.out.println("Interrupted! Clean up...");
        Thread.currentThread().interrupt(); // re-set flag
    }
});
sleeper.start();
sleeper.interrupt();   // sets interrupt flag → causes InterruptedException

Volatile & Visibility

volatile guarantees visibility (main memory flush) and prevents instruction reordering. Does NOT guarantee atomicity.

volatileJMMVisibility

⚡ Core Theory — Volatile & Java Memory Model (JMM)

  • The Problem (CPU Caching): Modern CPUs have L1/L2/L3 caches. Each core may keep a local copy of a variable. Without synchronization, Thread A's write may NEVER be seen by Thread B — even if it executes "after" the write.
  • volatile guarantees: Every write is flushed to main memory immediately. Every read fetches from main memory. This prevents stale cache reads.
  • Happens-Before: A write to a volatile field happens-before every subsequent read of that field. This is the JMM's formal guarantee — not just a runtime behavior.
  • Instruction Reordering: JVM/JIT and CPUs reorder instructions for optimization. volatile acts as a memory barrier, preventing reordering around it.
  • volatile does NOT give atomicity: count++ is 3 operations: read → increment → write. Another thread can interleave between any step. Use AtomicInteger or synchronized for compound ops.
  • When to use volatile: Simple flag written by one thread, read by many (stop flags). Double-checked locking singleton. Status/state variables with single writer.
  • Double-Checked Locking: Without volatile on the instance field, the JVM may see a partially constructed object due to reordering of the constructor call and the reference assignment.
💡 Interview: "volatile prevents visibility bugs and reordering, but NOT race conditions on read-modify-write. Use AtomicInteger for counters."
☕VolatileExample.java
// WITHOUT volatile: JVM may cache 'running' in CPU register.
// Reader thread might loop FOREVER even after running = false!
// WITH volatile: every read goes to main memory — always sees latest value.
private static volatile boolean running = true;

public static void main(String[] args) throws InterruptedException {
    Thread reader = new Thread(() -> {
        while (running) { /* busy spin */ }
        System.out.println("Stopped!");
    });
    reader.start();
    Thread.sleep(1000);
    running = false;   // Immediately visible to reader thread ✓
    reader.join();
}

// ─── volatile GUARANTEES ────────────────────────────────────
// ✓ Visibility: writes flushed to main memory immediately
// ✓ Ordering:  happens-before guarantee (prevents reordering)
// ✗ Atomicity: count++ is STILL a race condition!

// ─── Use AtomicInteger for compound (read-modify-write) ops ─
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();          // atomic: safe without locks
count.compareAndSet(5, 10);       // CAS: if curr==5, set 10
count.getAndUpdate(x -> x * 2);  // atomic compute

// ─── Double-Checked Locking (needs volatile!) ────────────────
private static volatile Singleton instance;

public static Singleton getInstance() {
    if (instance == null) {            // first check (no lock)
        synchronized (Singleton.class) {
            if (instance == null) {    // second check (with lock)
                instance = new Singleton();
            }
        }
    }
    return instance;
}

Synchronized

Intrinsic locking — synchronized method, block on this, static method, private lock object. wait()/notify()/notifyAll()

synchronizedwait/notifyMonitor

🔒 Core Theory — Synchronized & Monitor Lock

  • Monitor (Intrinsic Lock): Every Java object has a built-in monitor. Only one thread can hold it at a time. synchronized acquires this monitor on entry, releases on exit (even on exception).
  • synchronized method vs block: Method-level locks on this (or Class for static). Block-level lets you lock on any object and narrow the critical section for better throughput.
  • Static synchronized: Locks on the Class object — shared across all instances. Instance and static locks are INDEPENDENT.
  • Private lock object (best practice): Prevents external code from accidentally locking on this and causing deadlocks. Gives you full control over who can synchronize on the lock.
  • Reentrancy: Java monitors are reentrant — a thread that already holds a lock CAN re-acquire it without deadlocking. Hold count increments per re-entry.
  • wait() / notify() / notifyAll(): Must be called inside a synchronized block on the same object. wait() atomically releases the lock and suspends. notify() wakes one random waiter. notifyAll() wakes all — always use while loop to re-check condition.
  • Guaranteed: Atomicity (only one thread in critical section) + Visibility (all writes before lock release are visible to next lock acquirer).
💡 Interview: "synchronized guarantees mutual exclusion AND visibility. Always use a private lock object. Always use while(!condition) with wait()."
☕SynchronizationExample.java
// 1. Synchronized instance method — lock = 'this'
public synchronized void increment() { count++; }

// 2. Synchronized static method — lock = Class object
public static synchronized void staticMethod() { /* ... */ }

// 3. Synchronized block on 'this' — fine-grained
public void method() {
    // non-critical work here (no lock)
    synchronized (this) {
        count++;  // only critical section locked
    }
}

// 4. Private lock object — BEST PRACTICE
// Prevents external code from accidentally locking on 'this'
private final Object lock = new Object();
public void safeMethod() {
    synchronized (lock) { count++; }
}

// 5. Two independent locks for independent resources
private final Object lockA = new Object();
private final Object lockB = new Object();
public void opA() { synchronized (lockA) { /* ... */ } }
public void opB() { synchronized (lockB) { /* ... */ } }

// 6. wait() / notifyAll() — MUST be inside synchronized block!
synchronized (sharedObj) {
    // ALWAYS use while (not if) to guard against spurious wakeups
    while (!condition) {
        sharedObj.wait();       // releases lock and waits
    }
    // do work
    sharedObj.notifyAll();      // wake ALL waiting threads (safer than notify())
}

// 7. Reentrancy — same thread can re-acquire its own lock
public synchronized void outer() {
    inner(); // OK! Java locks are reentrant
}
public synchronized void inner() { /* same lock, no deadlock */ }

ReentrantLock

Explicit locking with tryLock, timeout, interruptible acquisition, fairness, queue inspection

ReentrantLocktryLockFairness

🔑 Core Theory — ReentrantLock vs synchronized

Why ReentrantLock?

  • tryLock(): Non-blocking — returns false immediately if lock unavailable. Prevents deadlocks.
  • tryLock(timeout): Wait up to N time units, then give up. Avoids indefinite blocking.
  • lockInterruptibly(): Thread can be interrupted while waiting for the lock — not possible with synchronized.
  • Fairness: new ReentrantLock(true) → longest-waiting thread gets lock. Prevents starvation but reduces throughput.
  • Multiple Conditions: lock.newCondition() creates separate wait-sets — much more precise than notifyAll().
  • Queue inspection: getQueueLength(), hasQueuedThreads() — useful for monitoring/debugging.

synchronized vs ReentrantLock

  • synchronized: Simpler syntax, auto-release, no timeout, not interruptible, unfair
  • ReentrantLock: Manual unlock() in finally required, supports timeout, interruptible, optional fairness, multiple conditions
  • Performance: In Java 6+, both have similar throughput. ReentrantLock wins under high contention with fair mode or when tryLock is needed.
  • Critical rule: ALWAYS unlock in a finally block or you risk permanently locking out all threads!
💡 Interview: "Use ReentrantLock when you need tryLock, timeout, interruptible waiting, fairness, or multiple Condition objects."
☕ReentrantLockExample.java
// fair=true → longest-waiting thread gets lock (less throughput, no starvation)
// fair=false (default) → any thread may get it (higher throughput)
ReentrantLock lock = new ReentrantLock(true);

// ─── Basic lock/unlock — ALWAYS unlock in finally! ───────────
lock.lock();
try {
    System.out.println("HoldCount: " + lock.getHoldCount());     // 1
    System.out.println("isHeld:    " + lock.isHeldByCurrentThread()); // true
    System.out.println("isLocked:  " + lock.isLocked());          // true
    // Re-entrant: same thread can lock again
    lock.lock();
    System.out.println("HoldCount: " + lock.getHoldCount());     // 2
    lock.unlock(); // inner unlock
} finally {
    lock.unlock(); // outer unlock — always in finally!
}

// ─── tryLock() — non-blocking, returns false immediately ─────
if (lock.tryLock()) {
    try { System.out.println("Got lock immediately!"); }
    finally { lock.unlock(); }
} else {
    System.out.println("Lock not available — skipping");
}

// ─── tryLock(timeout) — wait up to N time ─────────────────────
if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
    try { /* got lock within 500ms */ }
    finally { lock.unlock(); }
}

// ─── lockInterruptibly() — can abort while waiting ───────────
try {
    lock.lockInterruptibly();   // throws InterruptedException if interrupted
    try { /* critical section */ }
    finally { lock.unlock(); }
} catch (InterruptedException e) {
    System.out.println("Interrupted while waiting for lock!");
}

// ─── Queue inspection ─────────────────────────────────────────
System.out.println("isFair:          " + lock.isFair());
System.out.println("getQueueLength:  " + lock.getQueueLength());
System.out.println("hasQueuedThreads:" + lock.hasQueuedThreads());
System.out.println("hasQueuedThread: " + lock.hasQueuedThread(someThread));

ReadWriteLock & StampedLock

ReadWriteLock: concurrent reads OR exclusive write. StampedLock: adds optimistic reads for maximum read throughput.

ReadWriteLockStampedLockOptimistic

📖 Core Theory — ReadWriteLock & StampedLock

ReentrantReadWriteLock

  • Read lock (shared): Multiple threads can hold simultaneously. Ideal for read-heavy workloads (caches, config).
  • Write lock (exclusive): Only one thread at a time. Blocks ALL readers AND other writers.
  • Lock downgrade: You can downgrade write→read (acquire read while holding write, then release write). Upgrade (read→write) is NOT supported — causes deadlock.
  • Writer starvation risk: In read-heavy systems, writers may starve. Use fair=true or StampedLock to address.

StampedLock (Java 8+)

  • Optimistic read: tryOptimisticRead() returns a stamp WITHOUT acquiring any lock. Then validate with validate(stamp). If a write happened, fall back to a real read lock. Fastest for mostly-read scenarios.
  • Not reentrant: Unlike ReentrantReadWriteLock, StampedLock is NOT reentrant. Do NOT call it recursively — deadlock!
  • Stamp-based API: All lock/unlock methods use a long stamp token. Lose the stamp → can't unlock!
  • Lock views: asReadLock(), asWriteLock() provide Lock-compatible views for interop.
💡 Interview: "StampedLock's optimistic read is the fastest — zero blocking, just validate afterwards. Use when reads vastly outnumber writes."
☕ReadWriteStampedLock.java
// ─── ReentrantReadWriteLock ──────────────────────────────────
// Many threads can hold readLock simultaneously (no blocking between readers)
// writeLock is exclusive — blocks ALL readers AND writers
ReadWriteLock rwLock = new ReentrantReadWriteLock(true); // fair

public String read() {
    rwLock.readLock().lock();
    try {
        return data;    // concurrent reads allowed!
    } finally { rwLock.readLock().unlock(); }
}

public void write(String val) {
    rwLock.writeLock().lock();
    try {
        data = val;     // exclusive — no other reader or writer
    } finally { rwLock.writeLock().unlock(); }
}

// Inspection:
rwLock.getReadLockCount();              // number of read locks held
rwLock.isWriteLocked();                 // is write lock held?
rwLock.getWriteHoldCount();             // write lock hold count

// ─── StampedLock (Java 8+) ────────────────────────────────────
// Best feature: OPTIMISTIC READ — no lock acquired, just validate afterwards
StampedLock sl = new StampedLock();
double[] point = {3.0, 4.0};

public double distanceFromOrigin() {
    long stamp = sl.tryOptimisticRead();    // returns stamp, NO lock held
    double x = point[0], y = point[1];     // read without lock!
    if (!sl.validate(stamp)) {              // did a write happen?
        stamp = sl.readLock();              // fall back to real read lock
        try { x = point[0]; y = point[1]; }
        finally { sl.unlockRead(stamp); }
    }
    return Math.sqrt(x * x + y * y);
}

// Lock conversions:
long ws = sl.tryConvertToWriteLock(stamp);    // upgrade read → write
long rs = sl.tryConvertToReadLock(stamp);     // downgrade write → read
long os = sl.tryConvertToOptimisticRead(stamp);// release read, get optimistic

// Lock views (for API compatibility):
Lock readView  = sl.asReadLock();
Lock writeView = sl.asWriteLock();
ReadWriteLock rwView = sl.asReadWriteLock();

Condition Variables

Fine-grained signaling with ReentrantLock.newCondition() — separate conditions for producers and consumers

Conditionawait()signal()

🔔 Core Theory — Condition Variables

  • What is a Condition? A Condition is created from a ReentrantLock via lock.newCondition(). It is analogous to wait()/notify() but tied to an explicit lock, not an object monitor.
  • Multiple conditions (key advantage): With synchronized + notifyAll(), you wake ALL waiters and they all re-check — wasteful. With two Condition objects (notFull, notEmpty), producers only wake consumers and vice versa. Precise and efficient.
  • await(): Atomically releases the lock and suspends. Re-acquires the lock before returning. ALWAYS use inside a while loop (spurious wakeups!)
  • signal() vs signalAll(): signal() wakes ONE waiter on that specific Condition. signalAll() wakes ALL on that Condition. With single-consumer Conditions, signal() is safe and efficient.
  • awaitUninterruptibly(): Ignores interrupts while waiting — use when you must not abandon the wait.
  • Timed await: await(time, unit) and awaitNanos(ns) prevent deadlock by adding a timeout safety net.
💡 Interview: "Condition variables are the explicit-lock equivalent of wait/notify, but you can have MULTIPLE conditions per lock — producers and consumers each wait on their own queue."
☕ConditionExample.java
ReentrantLock lock = new ReentrantLock();
// Two SEPARATE conditions — much more precise than notifyAll()
Condition notFull  = lock.newCondition(); // "buffer has space"
Condition notEmpty = lock.newCondition(); // "buffer has items"

// ─── Producer ────────────────────────────────────────────────
public void produce(int item) throws InterruptedException {
    lock.lock();
    try {
        while (queue.size() == CAPACITY)
            notFull.await();       // wait ONLY on notFull (not all waiters!)
        queue.add(item);
        notEmpty.signal();         // wake exactly ONE consumer
    } finally { lock.unlock(); }
}

// ─── Consumer ────────────────────────────────────────────────
public int consume() throws InterruptedException {
    lock.lock();
    try {
        while (queue.isEmpty())
            notEmpty.await();      // wait ONLY on notEmpty
        int item = queue.poll();
        notFull.signal();          // wake exactly ONE producer
        return item;
    } finally { lock.unlock(); }
}

// ─── All Condition methods ───────────────────────────────────
notFull.await();                        // wait (releases lock)
notFull.awaitNanos(300_000_000L);       // timed wait (nanoseconds)
notFull.await(300, TimeUnit.MILLISECONDS); // timed wait
notFull.awaitUntil(deadline);           // wait until absolute date
notFull.awaitUninterruptibly();         // ignores Thread.interrupt()!
notFull.signal();                       // wake one waiter
notFull.signalAll();                    // wake all waiters

// Queue inspection:
lock.getWaitQueueLength(notFull);       // count of waiters on this condition
lock.hasWaiters(notFull);              // any thread waiting on it?

CountDownLatch

One-shot synchronizer: main thread waits for N worker threads to complete. Non-reusable.

CountDownLatchawait()countDown()

⏳ Core Theory — CountDownLatch

  • What it does: Initialized with count N. Worker threads call countDown() to decrement. Main thread calls await() and blocks until count reaches zero. Like a race starter waiting for all runners to check in.
  • One-shot (non-reusable): Once count reaches zero, it stays zero. All subsequent await() calls return immediately. Cannot be reset — for reusable barriers, use CyclicBarrier.
  • Any thread can countDown: Unlike locks, there is no ownership. Any thread (even threads not in the latch) can call countDown(). The count never goes below zero.
  • await() with timeout: Returns true if count reached zero in time, false if timed out. Use this in production to avoid indefinite blocking.
  • Common patterns: (1) Service startup — wait for N services to initialize before accepting requests. (2) Test synchronization — start all threads simultaneously. (3) Parallel computation — wait for all workers to finish before aggregating.
  • getCount(): Returns current count (long). Useful for diagnostics.
💡 Interview: "CountDownLatch = one-shot. CyclicBarrier = reusable. CountDownLatch: one thread waits for N. CyclicBarrier: all N wait for each other."
☕CountDownLatchExample.java
CountDownLatch latch = new CountDownLatch(3); // count = 3
System.out.println("Initial count: " + latch.getCount()); // 3

// Start 3 service threads
for (int i = 0; i < 3; i++) {
    final int id = i;
    new Thread(() -> {
        try {
            System.out.println("Service-" + id + " initializing...");
            Thread.sleep((long)(Math.random() * 2000));
            System.out.println("Service-" + id + " is UP!");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            latch.countDown(); // decrement count by 1
            System.out.println("Remaining: " + latch.getCount());
        }
    }).start();
}

// Main thread blocks until count reaches 0
latch.await();
System.out.println("All services ready! Starting application...");

// ─── await with timeout ───────────────────────────────────────
CountDownLatch timedLatch = new CountDownLatch(5);
boolean reachedZero = timedLatch.await(2, TimeUnit.SECONDS);
System.out.println("Reached zero: " + reachedZero); // false (only ~2s passed)
System.out.println("Count left:   " + timedLatch.getCount()); // 5

// ─── CountDownLatch API summary ───────────────────────────────
// countDown()              — decrement count (can be called by any thread)
// await()                  — block until count == 0
// await(timeout, unit)     — block with timeout, returns boolean
// getCount()               — current count (long)
// ⚠️ Cannot be reset! Use CyclicBarrier for reusable barriers.

CyclicBarrier

Reusable barrier: all N threads wait for each other, then proceed together. Optional barrier action runs when all arrive.

CyclicBarrierReusableBarrier Action

🚧 Core Theory — CyclicBarrier

  • What it does: All N parties call await(). The last to arrive triggers the optional barrier action (runs in that thread), then ALL are released simultaneously. Like a race where all runners must be at the start line before the gun fires.
  • Reusable (cyclic): After all parties are released, the barrier automatically resets for the next phase. This is the key difference from CountDownLatch.
  • Barrier action: Optional Runnable passed to the constructor. Runs exactly once per phase, in the last-arriving thread, BEFORE any thread proceeds. Perfect for aggregating phase results.
  • BrokenBarrierException: Thrown when a thread is interrupted while waiting, or when reset() is called while threads are waiting. Once broken, all waiting threads get this exception.
  • await() return value: Returns the arrival index (0 = last to arrive). You can use this to identify the "leader" thread per phase.
  • Use case: Parallel algorithms with phases (matrix multiplication, parallel sorting, multi-player game rounds where all players must finish a round before the next starts).
💡 Interview: "CyclicBarrier is reusable — use it for multi-round parallel algorithms. CountDownLatch is one-shot — use it for one-time initialization."
☕CyclicBarrierExample.java
// Barrier action runs ONCE when all parties arrive (in last-arriving thread)
CyclicBarrier barrier = new CyclicBarrier(3, () ->
    System.out.println("All arrived! Aggregating results...")
);

System.out.println("getParties(): " + barrier.getParties()); // 3

// CyclicBarrier IS reusable — workers can go through multiple rounds!
for (int round = 0; round < 2; round++) {
    for (int i = 0; i < 3; i++) {
        final int id = i;
        new Thread(() -> {
            try {
                System.out.println("Worker-" + id + " working...");
                Thread.sleep((long)(Math.random() * 500));

                // All 3 threads must call await() before any proceeds
                int arrivalPhase = barrier.await();
                System.out.println("Worker-" + id + " released from phase: " + arrivalPhase);
            } catch (InterruptedException | BrokenBarrierException e) {
                System.out.println("Barrier broken or interrupted!");
            }
        }).start();
    }
    Thread.sleep(1500); // wait for round to finish
}

// ─── await with timeout ───────────────────────────────────────
try {
    barrier.await(100, TimeUnit.MILLISECONDS); // TimeoutException if not all arrive
} catch (TimeoutException e) {
    System.out.println("Timeout! Barrier is now broken.");
}

// ─── CyclicBarrier API ─────────────────────────────────────────
barrier.getParties();        // total parties
barrier.getNumberWaiting();  // currently waiting count
barrier.isBroken();          // true if barrier disrupted
barrier.reset();             // reset to initial state (breaks waiting threads!)

Semaphore

Controls access to N concurrent slots — connection pools, rate limiting. Any thread can release (unlike locks).

Semaphoreacquire()Permits

🚦 Core Theory — Semaphore

  • What it is: A counting semaphore with N permits. acquire() decrements (blocks if 0). release() increments (unblocks a waiter). Conceptually: a bucket of N tokens.
  • Not a lock (no ownership): Any thread can call release(), even one that never called acquire(). This makes it suitable for producer-consumer handoff and can even allow permits to go ABOVE the initial count!
  • Mutex with Semaphore(1): A binary semaphore (new Semaphore(1)) acts like a mutex, but since there's no ownership, it can be released by a different thread — useful for inter-thread signaling.
  • Connection pool pattern: Initialize with N = max connections. Each thread acquires before using a connection, releases after. Automatically throttles concurrent DB access.
  • Rate limiting: Use with tryAcquire() to implement rate limiting — reject requests when all permits are taken.
  • drainPermits(): Acquires all available permits atomically. Useful for maintenance/shutdown scenarios.
  • Fairness: new Semaphore(n, true) ensures FIFO ordering — longest-waiting thread gets next permit. Default is unfair (higher throughput).
💡 Interview: "Semaphore controls CONCURRENCY (how many threads can run). Lock controls EXCLUSIVITY (only one thread). Semaphore(1) ≈ mutex but without ownership."

Phaser

Dynamic multi-phase barrier — threads can join/leave between phases. More flexible than CyclicBarrier.

PhaserMulti-PhaseDynamic

📋 Core Theory — Phaser

  • What it is: A generalized, reusable synchronization barrier that supports dynamic registration of parties and multiple phases. Think CyclicBarrier + dynamic parties + phase tracking.
  • Dynamic parties: Threads can register() or arriveAndDeregister() at any time. This is impossible with CyclicBarrier (fixed party count at construction).
  • arriveAndAwaitAdvance(): Arrive at current phase AND wait for all to arrive before advancing. Equivalent to CyclicBarrier.await() but with a phase number.
  • arrive(): Non-blocking arrival — marks arrival but does NOT wait. Useful when a thread wants to "check in" without blocking.
  • Phase number: Starts at 0, increments each time all parties arrive. getPhase() returns current phase. Returns negative when terminated.
  • Termination: Override onAdvance(phase, registeredParties) to define termination condition (e.g., after 5 phases). Return true from it to terminate the phaser.
  • Tiered phasers: Phasers can be organized in a tree hierarchy (parent-child) to manage thousands of parties efficiently without contention.
💡 Interview: "Phaser = CyclicBarrier + dynamic participants + multiple phases. Use when the number of threads changes across rounds or you need phase-based lifecycle control."

Exchanger

Synchronization point where exactly two threads swap objects. Blocks until the partner arrives.

Exchangerexchange()Two-Thread

🔄 Core Theory — Exchanger

  • What it is: A synchronization point for EXACTLY TWO threads to swap data. Each thread presents an object and receives the other's object. Both block until both have arrived.
  • Rendezvous point: Both threads must call exchange(). The first to arrive blocks; when the second arrives, both are unblocked simultaneously with each other's data.
  • exchange() with timeout: Throws TimeoutException if the partner doesn't arrive within the given time. Critical for preventing deadlocks when the partner thread may have crashed.
  • Double-buffering pattern: Thread A fills Buffer1 while Thread B drains Buffer2. When both finish, they swap via Exchanger. Thread A now drains Buffer1 (filled by B's previous round). Zero idle time, maximum throughput.
  • Only two threads: If more than 2 threads use the same Exchanger, the pairing is arbitrary and non-deterministic. Design for exactly 2 partners.
  • Use cases: Pipeline stages, producer-consumer with backpressure, genetic algorithm crossover, network packet I/O buffering.
💡 Interview: "Exchanger is for 2-thread data swapping. Classic use: double-buffering where one thread fills while the other drains, then they swap buffers."

ExecutorService

All thread pool types, submit, invokeAll, invokeAny, Future methods, and proper shutdown lifecycle.

ExecutorServiceThreadPoolFutureinvokeAll

⚙️ Core Theory — ExecutorService & Thread Pools

Pool Types

  • FixedThreadPool(n): n threads always alive. Bounded concurrency. Good for CPU-bound tasks. Queue is unbounded (risk: OOM if tasks pile up).
  • CachedThreadPool: Grows unboundedly. Threads idle for 60s then die. Good for short-lived I/O tasks. Risk: thread explosion.
  • SingleThreadExecutor: 1 thread, sequential execution, ordered FIFO. If thread dies, a new one replaces it.
  • ScheduledThreadPool: For delayed and periodic tasks. Use instead of Timer (which stops on uncaught exception).
  • WorkStealingPool: ForkJoinPool under the hood. Threads steal work from other threads' queues. Best for recursive/divide-and-conquer tasks.

ThreadPoolExecutor params

  • corePoolSize: Minimum threads always alive (even idle)
  • maximumPoolSize: Max threads. Extra threads created only when queue is full.
  • keepAliveTime: Idle non-core threads die after this duration.
  • workQueue: Bounded = backpressure. Unbounded = OOM risk. SynchronousQueue = no buffering.
  • RejectionPolicy: AbortPolicy (throw), CallerRunsPolicy (run in caller), DiscardPolicy (drop silently), DiscardOldestPolicy (drop oldest).
  • Shutdown: shutdown() stops new tasks, drains queue. shutdownNow() interrupts running tasks, returns unstarted list.
💡 Interview: "Never use newCachedThreadPool() in production for unbounded workloads. Always use bounded ThreadPoolExecutor with a rejection handler to prevent resource exhaustion."

ScheduledExecutorService

One-shot delay, fixed-rate repetition, and fixed-delay repetition with cancellable ScheduledFuture.

ScheduledFixedRateFixedDelay

🕰️ Core Theory — ScheduledExecutorService

  • schedule(): Run a task ONCE after a specified delay. Returns a ScheduledFuture which can be cancelled or inspected.
  • scheduleAtFixedRate(task, initialDelay, period): Delay between the START of consecutive executions. If a task takes longer than the period, the next run starts immediately after (no overlap, tasks run sequentially). Good for heartbeats, polls.
  • scheduleWithFixedDelay(task, initialDelay, delay): Delay between the END of one execution and the START of the next. Better when task duration is variable and you need breathing room between runs.
  • vs Timer: Timer is single-threaded — one delayed task blocks all others. An uncaught exception kills the timer permanently. ScheduledExecutorService is multi-threaded and more robust.
  • Exception handling: If a recurring task throws an uncaught exception, the future silently stops recurring! Always wrap task body in try-catch.
  • ScheduledFuture.getDelay(): Returns remaining delay. Negative means overdue.
  • cancel(false): Don't interrupt if running. cancel(true): interrupt if running. For recurring tasks, cancels all future executions.
💡 Interview: "FixedRate = constant frequency regardless of task duration. FixedDelay = constant gap AFTER task completes. Always wrap periodic task bodies in try-catch to prevent silent death."

CompletableFuture

Non-blocking async pipelines — creation, chaining, combining, error handling, and manual completion.

CompletableFutureAsync PipelineJava 8+allOf/anyOf

⚡ Core Theory — CompletableFuture

Key Concepts

  • Non-blocking chains: Unlike Future.get() (always blocks), CompletableFuture chains callbacks that run when the result is ready. True async composition.
  • thenApply vs thenCompose: thenApply transforms the result (like map). thenCompose chains CFs avoiding nesting (like flatMap). Use thenCompose when the next step returns a CF.
  • thenAccept vs thenRun: thenAccept gets the result (Consumer). thenRun gets no result (Runnable).
  • Async variants (thenApplyAsync): Without Async = runs in the same thread that completed the CF. With Async = submitted to ForkJoinPool.commonPool() or your executor.
  • allOf vs anyOf: allOf completes when ALL complete (returns Void CF — you must get each result individually). anyOf completes when the FIRST completes.

Error Handling

  • exceptionally(fn): Only called on failure. Recovery function. Like catch block — chain continues with recovered value.
  • handle(BiFunction): Always called (result or exception). Like try-finally. Returns a new value.
  • whenComplete(BiConsumer): Always called, but result passes through unchanged. Side effects only.
  • get() vs join(): get() throws checked exceptions (InterruptedException, ExecutionException). join() throws unchecked. Prefer join() in lambda chains.
  • CompletionException: Wraps the original exception in join(). Unwrap with getCause().
💡 Interview: "thenCompose = flatMap (avoids CF nesting). thenApply = map. handle() is always called. exceptionally() only on error. allOf returns Void — call .get() on each individual CF for results."

Producer-Consumer Patterns

Classic wait/notifyAll, Lock+Condition (two separate), BlockingQueue (modern), LinkedTransferQueue (direct handoff)

Producer-ConsumerBlockingQueuePattern

🟭 Core Theory — Producer-Consumer Patterns

  • Why Producer-Consumer? Decouples producers (data generators) from consumers (data processors). Buffer absorbs speed mismatches. Enables pipelining and backpressure.
  • Pattern 1 (wait/notifyAll): Classic but error-prone. Must use while (not if) for spurious wakeups. notifyAll() wakes both producers AND consumers (wasteful).
  • Pattern 2 (Lock + 2 Conditions): notFull condition for producers, notEmpty for consumers. signal() (not signalAll) wakes exactly one correct waiter. More efficient.
  • Pattern 3 (BlockingQueue): Cleanest — handles all synchronization internally. put() blocks if full. take() blocks if empty. Just use it!
  • SynchronousQueue: Zero capacity. put() blocks until a consumer calls take(). Direct handoff — producer and consumer must be simultaneously present. Used inside newCachedThreadPool()!
  • LinkedTransferQueue: Like SynchronousQueue + queue. transfer() = direct handoff if consumer waiting, else enqueue. tryTransfer() = only if consumer immediately available.
  • PriorityBlockingQueue: Unbounded queue where elements are taken in priority order (natural ordering or Comparator). Always use bounded alternatives to avoid OOM.
💡 Interview: "Use BlockingQueue in production. Avoid manual wait/notify — it's error-prone. SynchronousQueue is used inside CachedThreadPool for zero-buffer direct task handoff."
☕ProducerConsumerPatterns.java
// ─── Pattern 1: synchronized + wait/notifyAll ─────────────────
class Buffer {
    private final Queue<Integer> q = new LinkedList<>();
    private final int CAPACITY;

    public synchronized void produce(int val) throws InterruptedException {
        while (q.size() == CAPACITY) wait();  // ALWAYS while loop (spurious wakeups!)
        q.add(val);
        notifyAll();   // wake consumers
    }
    public synchronized int consume() throws InterruptedException {
        while (q.isEmpty()) wait();            // ALWAYS while loop
        int val = q.poll();
        notifyAll();   // wake producers
        return val;
    }
}

// ─── Pattern 2: ReentrantLock + 2 Conditions (more precise) ───
ReentrantLock lock = new ReentrantLock();
Condition notFull  = lock.newCondition();
Condition notEmpty = lock.newCondition();

// Producer:
lock.lock(); try {
    while (q.size() == CAP) notFull.await();  // wait ONLY for space
    q.add(val); notEmpty.signal();             // wake only ONE consumer
} finally { lock.unlock(); }

// Consumer:
lock.lock(); try {
    while (q.isEmpty()) notEmpty.await();      // wait ONLY for item
    int val = q.poll(); notFull.signal();      // wake only ONE producer
} finally { lock.unlock(); }

// ─── Pattern 3: BlockingQueue (cleanest — handles sync internally) ─
BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(10);
// Producer:  queue.put(item);     // blocks if full
// Consumer:  queue.take();        // blocks if empty

// ─── Pattern 4: LinkedTransferQueue (direct handoff) ──────────
LinkedTransferQueue<String> ltq = new LinkedTransferQueue<>();

// Producer: transfer() blocks until a consumer actually takes it
ltq.transfer("direct-message");         // guaranteed rendezvous
ltq.tryTransfer("try");                 // non-blocking: false if no consumer
ltq.tryTransfer("timed", 300, TimeUnit.MILLISECONDS); // timed
ltq.getWaitingConsumerCount();          // consumers blocked in take()
ltq.hasWaitingConsumer();

pbq.put(new Task("Medium", 2));

ConcurrentHashMap

Lock-free reads, per-bin locking on writes, atomic compute/merge operations, bulk parallel operations.

ConcurrentHashMapAtomic OpsLock-Free

🗺️ Core Theory — ConcurrentHashMap

  • Java 8+ internal structure: Array of bins (buckets). Short chains use linked lists. Long chains (8+ entries) convert to balanced trees (TreeBin) for O(log n) lookup.
  • Reads are lock-free: Node values and next pointers are volatile. Reads use no locks — extremely fast and non-blocking.
  • Writes use per-bin locking: Only the single affected bin (head node) is synchronized. Other bins are unaffected. This gives ~16x better write concurrency vs Hashtable.
  • CAS for empty bins: Inserting into an empty bin uses Compare-And-Swap — zero contention, no locking at all.
  • compute() is atomic: The entire function (read + compute + write) is atomic per bin. Use it instead of get+put to avoid TOCTOU races.
  • null not allowed: Both keys and values must be non-null. Reason: in concurrent contexts, you can't distinguish "key absent" from "key mapped to null" without additional locking.
  • size() is approximate: Under concurrent modification, size() may not be exact. Use mappingCount() (returns long) for large maps. For exact counts, use LongAdder.
💡 Interview: "CHM reads are lock-free. Writes lock only the single bin. compute() and merge() are atomic. Never use null keys/values. Use mappingCount() over size() for large maps."
☕ConcurrentHashMapExample.java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// ─── Atomic Operations (all guaranteed thread-safe) ───────────
map.putIfAbsent("key", 1);                    // put only if absent
map.getOrDefault("key", -1);                  // get or default
map.remove("key", 1);                         // conditional remove

// computeIfAbsent — create if not present (atomic)
map.computeIfAbsent("key", k -> expensiveInit(k));

// computeIfPresent — update if present (atomic)
map.computeIfPresent("key", (k, v) -> v + 1);

// compute — always applies function, null return removes key
map.compute("key", (k, v) -> v == null ? 1 : v + 1);

// merge — combine new value with existing using BiFunction
map.merge("key", 1, Integer::sum);            // 1 if absent, sum if present

// replace — conditional update
map.replace("key", oldVal, newVal);           // atomic CAS-like replace

// ─── Bulk Operations (parallelismThreshold = 1 → always parallel) ─
map.forEach(1, (k, v) -> System.out.println(k + "=" + v));
String found = map.search(1, (k, v) -> v > 100 ? k : null); // first non-null
Integer total = map.reduce(1, (k, v) -> v, Integer::sum);    // map + reduce
Integer maxVal = map.reduceValues(1, Integer::max);
String keys = map.reduceKeys(1, (a, b) -> a + "," + b);

// ─── Inspection ───────────────────────────────────────────────
map.mappingCount();           // long (better than size() for huge maps)
map.size();                   // int version
map.isEmpty();
map.containsKey("key");
map.containsValue(42);

BlockingQueue Types

📦 Core Theory — BlockingQueue Types

Queue Comparison

  • ArrayBlockingQueue: Bounded, array-backed. Optional fairness. Fixed capacity. Best when you want backpressure and bounded memory.
  • LinkedBlockingQueue: Optionally bounded (default unbounded = Integer.MAX_VALUE). Separate head/tail locks for producer and consumer — higher throughput than ABQ under high concurrency.
  • PriorityBlockingQueue: Unbounded. Min-heap ordering. Elements MUST be Comparable or supply Comparator. No fairness guarantee among equal-priority elements.
  • SynchronousQueue: Zero capacity. Direct handoff. Put blocks until take, take blocks until put. Used in newCachedThreadPool().
  • DelayQueue: Elements only available after their delay expires. Elements implement Delayed interface. Used for scheduled task queues, caches with TTL expiration.
  • LinkedBlockingDeque: Double-ended blocking queue. Elements can be added/removed from both ends. Useful for work-stealing algorithms.

Operation Behavior

  • put(e) / take(): Block indefinitely (interrupt-capable). Use in producer/consumer threads.
  • offer(e) / poll(): Return immediately (false/null if full/empty). Use for non-blocking checks.
  • offer(e, t, u) / poll(t, u): Timed blocking. Best for production code — avoids permanent blocking.
  • peek(): View head without removing. Non-blocking, returns null if empty.
  • drainTo(Collection): Atomically moves all available elements to collection. Efficient batch processing.
  • remainingCapacity(): For bounded queues only. Returns available space. Useful for backpressure monitoring.
💡 Interview: "Always prefer bounded queues in production to prevent OOM. Use put()/take() for blocking semantics, offer(timeout)/poll(timeout) for resilient code."

Atomic Variables

Lock-free CAS operations: AtomicInteger, AtomicLong, AtomicReference, AtomicArray, LongAdder, LongAccumulator

AtomicCASLock-FreeLongAdder

⚛️ Core Theory — Atomic Variables & CAS

  • CAS (Compare-And-Swap): The CPU instruction behind all atomic operations. "If current value == expected, atomically swap to new value. Otherwise fail and return the actual value." Zero blocking, hardware-level atomicity.
  • ABA Problem: Thread reads A, another changes A→B→A, CAS succeeds despite intermediate change. Solution: AtomicStampedReference (adds version counter) or AtomicMarkableReference (adds boolean mark).
  • compareAndSet vs compareAndExchange (Java 9): compareAndSet returns boolean (success/failure). compareAndExchange returns the witness value (actual current value) — easier retry loops.
  • LongAdder vs AtomicLong: Under high contention, multiple threads CAS-ing the same AtomicLong cause cache-line bouncing (each CAS failure retries). LongAdder uses striped cells — each thread has its own cell, reducing contention. sum() adds all cells. ~10x faster under high contention but slightly approximate during concurrent updates.
  • AtomicReference: CAS on object references. Use for lock-free data structures (linked lists, stacks). Be careful of ABA problem with mutable objects.
  • VarHandle (Java 9+): Low-level variable access with memory ordering semantics. More powerful than Atomic classes. Used internally by java.util.concurrent.
💡 Interview: "CAS = atomic read-compare-write in one CPU instruction. LongAdder beats AtomicLong at high contention by reducing CAS collisions via thread-local striping. AtomicStampedReference fixes ABA."

🎤 Interview Strategy

  • Structure your answer: (1) Define the concept. (2) Explain the problem it solves. (3) Give a concrete use case. (4) Mention gotchas/edge cases. (5) Compare alternatives.
  • Golden rules to always mention: Use while not if with wait(). Always unlock in finally. Always call threadLocal.remove() in thread pools. Prefer BlockingQueue over manual wait/notify.
  • Know your guarantees: volatile = visibility + ordering. synchronized = visibility + atomicity + mutual exclusion. AtomicXxx = lock-free atomicity via CAS.
Q1Process vs Thread — what's the difference?▼

Process: independent execution unit with its own memory space, file handles, and resources. Processes are isolated — one crash doesn't affect others.

Thread: lightweight unit of execution within a process. Threads share heap memory, static fields, and open files. Context switching between threads is ~10x cheaper than between processes.

Key: Multiple threads share the heap but each has its own stack, PC register, and local variables.

Q2volatile vs synchronized — when to use which?▼

volatile: guarantees visibility + ordering. Does NOT guarantee atomicity. Use for simple flags/booleans written by one thread and read by many (e.g., stopFlag, double-checked locking).

synchronized: guarantees visibility + atomicity + mutual exclusion. Use for compound operations like count++, check-then-act, or any multi-step critical section.

Rule: if the operation is a single write/read, volatile may suffice. If it's read-modify-write, use synchronized or AtomicInteger.

Q3Why while loop instead of if with wait()?▼

Spurious wakeups: The Java specification allows threads to wake from wait() without notify() being called. This is a hardware/OS phenomenon.

Using if (!condition) wait() means a spurious wakeup skips the check — the thread proceeds even when the condition is false, causing bugs.

Using while (!condition) wait() re-checks after every wakeup. If the condition is still false, the thread goes back to sleep. Always use while.

Q4notify() vs notifyAll() — which to use?▼

notify(): wakes ONE arbitrary waiting thread. Risk: the wrong thread wakes up and goes back to sleep, causing missed signals and potential deadlock.

notifyAll(): wakes ALL waiting threads. They re-check their conditions; only those whose condition is now true proceed. Safer but more overhead.

Prefer notifyAll() unless: (1) all waiters are identical and (2) only one needs to proceed. Use two Condition objects with ReentrantLock for fine-grained signaling.

Q5How to prevent deadlock?▼

Deadlock needs all 4 Coffman conditions: mutual exclusion, hold-and-wait, no preemption, circular wait. Break any one:

1. Lock ordering: always acquire locks in the same global order across all threads.
2. tryLock(timeout): back off if lock unavailable — no indefinite blocking.
3. Avoid nested locks: don't hold one lock while acquiring another.
4. Use abstractions: BlockingQueue, Semaphore avoid manual lock management.
5. Lock timeout + retry: with ReentrantLock.tryLock(timeout).

Q7What is ThreadLocal and when to use it?▼

ThreadLocal gives each thread its own independent copy of a variable. Changes in one thread are never visible to others — no synchronization needed.

Use cases: per-request DB connections, security context, SimpleDateFormat (not thread-safe), transaction IDs, user sessions in web frameworks (Spring Security uses it!).

⚠️ Critical: Always call threadLocal.remove() after use in thread pools! The thread is reused for the next task — leftover values cause subtle bugs and memory leaks.

Q8How does ConcurrentHashMap achieve thread-safety?▼

Java 8+ implementation:

• Reads: completely lock-free using volatile fields and CAS
• Writes: use synchronized on the individual bin (hash bucket), not the whole map
• CAS: Compare-And-Swap for inserting into empty bins
• 16+ bins by default: only 1/16th of the map is locked per write

vs Hashtable: locks the ENTIRE map on every operation → huge contention.
vs Collections.synchronizedMap(): same — single global lock.

Q10LongAdder vs AtomicLong — when to use which?▼

AtomicLong: single CAS cell, simple. Best for low-to-medium contention or when you need the exact current value (get()) or non-increment operations.

LongAdder: uses striped cells — each contending thread has its own cell. sum() combines all cells. Best for high-contention counters (metrics, statistics). Much faster under heavy load.

Rule: metrics/counters → LongAdder. Need CAS or exact reads → AtomicLong.

Q16When should you use Virtual Threads vs Platform Threads?▼

Virtual threads are ideal when threads spend most of their time blocking on I/O (database queries, HTTP calls, file reads). In these cases, a single JVM can handle millions of concurrent requests with the same throughput as a reactive framework — but with simpler, synchronous-looking code.

Platform threads are still better for CPU-bound tasks (number crunching, encoding, compression). Virtual threads give no throughput benefit here since the CPU is always busy.

Q11What is a race condition and how do you prevent it?▼

Race condition: When program correctness depends on the relative timing/ordering of thread execution. Result is unpredictable and often non-reproducible.

Classic example: count++ — this is read-increment-write. Two threads can read the same value, both increment, both write, losing one increment.

Prevention strategies:
1. synchronized: mutual exclusion — only one thread at a time
2. AtomicInteger: CAS-based lock-free update for simple counters
3. Immutable objects: if state can't change, no race is possible
4. Thread confinement: ThreadLocal — each thread has its own copy
5. Proper happens-before: volatile, locks, or concurrent collections that provide ordering guarantees

Q12What is the Java Memory Model (JMM) and happens-before?▼

The JMM defines rules for how threads interact through shared memory. Without it, CPUs and JIT compilers can reorder instructions in ways that break multithreaded code.

Happens-Before (HB): If action A HB action B, then A's results are guaranteed visible to B. HB is transitive — A HB B, B HB C ⟹ A HB C.

HB relationships:
• A volatile write HB every subsequent read of that field
• unlock() HB every subsequent lock() of the same monitor
• thread.start() HB every action in the started thread
• thread.join(): all actions in thread HB after join() returns
• Static initializer completes HB first access to that class

Without HB, even if code looks "correct" sequentially, the JVM may reorder writes/reads causing visibility bugs.

Q14How does ForkJoinPool differ from a regular thread pool?▼

ForkJoinPool (Java 7+) is designed for divide-and-conquer algorithms using the fork-join framework.

Work Stealing: Each worker thread has its own deque (double-ended queue) of tasks. When a thread finishes its work, it steals tasks from the tail of other threads' deques. This maximizes CPU utilization without central contention.

Key differences from ThreadPoolExecutor:
• ForkJoinPool: M worker threads, each with own deque + work stealing. Best for recursive, CPU-bound tasks that split into subtasks.
• ThreadPoolExecutor: N threads sharing one queue. Best for independent tasks (I/O, web requests).

RecursiveTask (returns value) and RecursiveAction (void) are the ForkJoinPool task types. Use fork() to submit subtask, join() to wait for result.

ForkJoinPool.commonPool() is shared by CompletableFuture, parallel streams, and other frameworks.

🚀 Core Theory — Virtual Threads (Project Loom)

Platform vs Virtual Threads

  • Platform thread: 1:1 mapping with an OS thread. ~1 MB stack. Expensive to create. Max practical ~thousands per JVM.
  • Virtual thread: JVM-managed. Mounted onto a carrier (platform) thread from a ForkJoinPool. ~few KB of memory. Millions can coexist.
  • M:N threading: M virtual threads multiplexed onto N carrier threads (N ≈ CPU cores). When a virtual thread blocks on I/O, its carrier is freed to run another virtual thread.
  • Always daemon: Virtual threads are always daemon threads. Setting setDaemon(false) throws IllegalArgumentException.
  • Thread-local works: ThreadLocal works but can cause memory pressure at scale (millions of copies). Prefer ScopedValue (Java 21 preview) instead.

Key Rules & Gotchas

  • Ideal for I/O-bound work: Virtual threads shine when threads spend most time waiting (DB, HTTP, file). For CPU-bound work, they give no benefit over platform threads.
  • Pinning: A virtual thread is pinned to its carrier when inside a synchronized block or native method. The carrier cannot be freed during pinning. Use ReentrantLock instead of synchronized for lock-heavy virtual thread code.
  • Thread pool anti-pattern: Do NOT use a fixed pool of virtual threads — defeats the purpose. Create one-per-task with newVirtualThreadPerTaskExecutor().
  • Structured Concurrency (Java 21 preview): StructuredTaskScope ensures subtask lifetimes are bounded by the parent scope. Prevents orphan threads. Use ShutdownOnFailure or ShutdownOnSuccess policies.
  • ScopedValue (Java 21 preview): Immutable per-scope data, safer and more efficient than ThreadLocal for virtual threads at scale.
